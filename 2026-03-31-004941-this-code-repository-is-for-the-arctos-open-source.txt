⏺ Here's a walkthrough of convertJoint.py:

  Gear Ratios

  The actual ARCTOS gear ratios replace the placeholder values from convert.py:
  - J1: 13.5 (belt), J2/J3: 150 (cycloidal), J4: 48 (planetary), J5/J6: 67.82
  (planetary)

  Core Logic: joint_to_motor_positions()

  This is the key new function. It takes 6 joint angles (from the G-code) and
  returns 6 motor positions:

  Joints 1–4 — straightforward:
  motor_pos = joint_angle × gear_ratio

  Joints 5–6 — differential coupling:
  Motor_B = (q5 + q6) × 67.82
  Motor_C = (q5 - q6) × 67.82

  This means if you want only J5 to move (q5=90, q6=0), both motors turn the
  same amount in the same direction. If you want only J6 to move (q5=0, q6=90),
  the motors turn equal amounts in opposite directions. The differential
  mechanically combines these into independent wrist pitch and roll.

  After computing motor positions, direction inversion is applied to J1 and J2
  (negating their values).

  CAN Message Generation: convert_to_can_message()

  Simplified from convert.py — it takes a motor position directly (gear ratio
  and coupling already applied) instead of a raw joint angle + gear ratio. It:
  1. Computes the relative position from the last commanded motor position (for
  incremental movement)
  2. Encodes it as a signed 24-bit hex value (two's complement)
  3. Formats the CAN message: [axis_id][F5][speed][02][position]

  Data Flow

  test.tap (joint angles)
      → parse G-code (F speed, G90 X Y Z A B C)
      → joint_to_motor_positions() applies gear ratios + differential coupling +
   inversion
      → convert_to_can_message() generates hex CAN frames
      → calculate_crc() appends checksum
      → test.txt (CAN messages, compatible with send.py)

  What's different from convert.py

  1. Correct gear ratios instead of placeholders
  2. Differential coupling for J5/J6 — convert.py treats all joints
  independently
  3. Cleaner separation — gear ratio math happens once in
  joint_to_motor_positions() before CAN encoding, rather than being mixed into
  the CAN message function

⏺ Inputs: test.tap (G-code)

  F1000                              ← motor speed (shared by all axes)
  G90 X0 Y0 Z0 A0 B10 C0            ← joint angles in degrees

  - F — speed value passed through to the CAN message
  - G90 — absolute positioning mode flag (triggers parsing)
  - X, Y, Z, A, B, C — desired joint angles (degrees) for joints 1–6 of the
  robot arm

  Each G90 line is one waypoint — the robot moves all 6 joints simultaneously to
   reach that pose.

  Processing: convertJoint.py

  Joint angles go through three transformations:

  1. Gear ratios — multiply by the mechanical reduction (e.g., 1° on J2 = 150°
  on motor 2)
  2. Differential coupling — J5/J6 are combined: M5 = (B+C)×67.82, M6 =
  (B-C)×67.82
  3. Direction inversion — J1 and J2 are negated

  Output: test.txt (CAN messages)

  Each G90 line produces 6 CAN messages (one per motor), written as hex strings:

  01F503E80200053DB4
  │ │  │   │  │    └─ CRC (1 byte)
  │ │  │   │  └───── relative position × 100, signed 24-bit hex
  │ │  │   └──────── command byte (02 = position mode)
  │ │  └──────────── speed (F1000 = 0x03E8)
  │ └─────────────── message type (F5 = motion command)
  └───────────────── motor ID (01–06)

  The position is relative — it's the difference from the last commanded motor
  position, scaled by ×100 for precision. This is what send.py transmits over
  CAN to the motor controllers.

❯ explain how it works                                                          

⏺ generate_trajectory.py creates a .tap file with smooth, coordinated multi-axis
   motion by computing waypoints along parametric curves.

  How it works

  A parameter t sweeps from 0.0 to 1.0 across steps waypoints. Each axis is a
  function of t:

  - X — linear interpolation from -45° to +45°. Sweeps the base joint left to
  right at constant angular rate.
  - Y — sin(2πt) with 10° amplitude. One full sine cycle, so the shoulder goes
  up, back to center, down, back to center during the sweep.
  - Z — sin(πt) with 5° amplitude. Half a sine cycle, so the elbow gently lifts
  mid-sweep and returns to zero at both ends.
  - A, B, C — held at 0° (wrist stays fixed).

  Each computed pose becomes one G90 line in the .tap file. A final G90 X0 Y0 Z0
   A0 B0 C0 returns to home.

  Parameters

  ┌─────────────┬───────────┬───────────────────────────────────────┐
  │    Param    │  Default  │               Controls                │
  ├─────────────┼───────────┼───────────────────────────────────────┤
  │ steps       │ 50        │ Number of waypoints (more = smoother) │
  ├─────────────┼───────────┼───────────────────────────────────────┤
  │ speed       │ 1000      │ F value in G-code                     │
  ├─────────────┼───────────┼───────────────────────────────────────┤
  │ x_range     │ (-45, 45) │ Sweep limits for X                    │
  ├─────────────┼───────────┼───────────────────────────────────────┤
  │ y_amplitude │ 10°       │ How far Y oscillates                  │
  ├─────────────┼───────────┼───────────────────────────────────────┤
  │ z_amplitude │ 5°        │ How far Z oscillates                  │
  └─────────────┴───────────┴───────────────────────────────────────┘

  Pipeline

  generate_trajectory.py → test.tap → convertJoint.py → test.txt → send.py →
  robot

  You can tweak any axis function — swap sin for a linear ramp, add motion to
  B/C for wrist movement, change frequencies, etc. Want me to adjust any of the
  motion profiles?

  convertJoint.py follows the same pattern — last_motor_positions tracks state
  so it can compute the relative change for each CAN frame.
